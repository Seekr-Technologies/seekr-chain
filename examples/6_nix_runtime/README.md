# Nix runtime example

Run a job whose Python environment is a nix closure instead of a Docker image.

```
flake.nix    →  python 3.12 + requests   (the runtime environment)
job.py       →  the script that runs inside that environment
config.yaml  →  glue: tells seekr-chain to fetch the closure + run job.py
```

The container the cluster pulls is a small nix-runner bootstrap (~250 MB
with `nix` + `s5cmd`). The actual runtime is fetched from your binary cache
as a content-addressed closure at pod startup. Adding a Python package
later is an edit to `flake.nix` and a re-submit — no docker image rebuild,
no docker layer caching to think about.

## What seekr-chain does on submit

1. **Local eval** — `nix eval --raw ./#packages.x86_64-linux.default.outPath`
   resolves the closure hash without building anything (eval is pure, runs
   on any system, including aarch64 Macs).
2. **Cache check** — HEAD on `<nix_store>/<hash>.narinfo`.
3. **If present** → single-step workflow: the user step pulls the closure
   from the cache at pod startup and runs `python job.py`.
4. **If missing** → seekr-chain auto-injects an in-cluster build step that
   runs `nix build` + `nix copy --to <store>`. The user step `depends_on`
   the build step.

First submit: a build step compiles the closure once. Every subsequent
submit (until you change `flake.nix`) skips the build entirely.

## One-time setup

In `~/.seekrchain.toml` (or via env vars):

```toml
nix_store         = "s3://my-bucket/nix-cache"
nix_runner_image  = "registry.example.com/seekr-chain-nix-runner:latest"
```

You also need `nix` on the submit machine for the local eval step.
Install from https://nixos.org/download with
`experimental-features = nix-command flakes` enabled.

If you don't want nix locally (or this is a CI runner), use `nix.closure:`
with an explicit `/nix/store/<hash>-<name>` path and set `build: false` —
seekr-chain will skip both the eval and the build, and fail at submit if
the closure isn't already in the cache.

## Run it

```sh
cd examples/6_nix_runtime
chain submit config.yaml --follow
```

First-run output (build step + user step in sequence):

```
[_nix_build_<hash>] === nix build path:./#packages.x86_64-linux.default ===
[_nix_build_<hash>] built closure: /nix/store/...-seekr-chain-example-nix-env
[_nix_build_<hash>] === nix copy --to s3://my-bucket/nix-cache ===
[step]              ============================================================
[step]              python  exe : /nix/store/.../bin/python3.12
[step]              python  ver : 3.12.7
[step]              requests ver: 2.31.0
[step]              ============================================================
[step]              Fetching https://httpbin.org/ip ...
[step]                origin: <pod's outbound IP>
[step]              Done.
```

Second-run output (no build step, the user step runs immediately):

```
[step]              ============================================================
[step]              python  exe : /nix/store/.../bin/python3.12
[step]              ...
```

## Extending the environment

Edit `flake.nix` to add packages from nixpkgs:

```nix
python = pkgs.python312.withPackages (ps: with ps; [
  requests
  rich        # ← added
]);
```

Then re-submit. The closure hash changes, so seekr-chain triggers one new
build step that uploads just the delta (the new `rich` store path plus a
regenerated python-env wrapper). All the unchanged paths in the closure
(glibc, python interpreter, requests, etc.) stay in the cache — no
rebuild, no re-upload.

This is the "thin top layer over a shared base" property that docker
layer caching gives you when you set up the Dockerfile carefully — except
here it falls out of the nix dependency graph automatically, with no
layer ordering to maintain.

## Files

| File | Purpose |
|---|---|
| `flake.nix` | Defines the closure: python 3.12 + requests + coreutils + bash. Pin to nixpkgs nixos-24.05 for reproducibility. |
| `flake.lock` | Generated by `nix flake lock` — pins the exact nixpkgs commit. Commit this to source control. |
| `job.py` | The script run inside the closure. Imports `requests`, fetches a URL, prints the result. |
| `config.yaml` | seekr-chain workflow: one step using `nix.expression: ./`, running `python job.py`. |
| `README.md` | This file. |
