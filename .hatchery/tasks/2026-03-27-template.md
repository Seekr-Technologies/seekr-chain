## Objective

Our code for creating/launching argo workflows has sprawled quite a lot.

At the heart, the only thing that it does is:

- Package and upload code
- Create screts
- Constructe kubernetes manifest
- Apply manifest


When I initially wrote this, I was trying to go as functional as possible - everything is a function. However, this makes understanding the actual construction of the manifest very difficult.

Instead, what do you think about using a template? Jinja, etc? Do you think we could use that to construct a significantly simpler process for building the manifest?

I want to keep an eye to towards ease of maintence and adding new features.

## Summary

Replaced the deeply nested Python dict manifest construction with Jinja2 YAML templates, making the Kubernetes manifest structure directly visible without tracing through chains of helper functions.

**Key decisions:**

- **Two templates**: `jobset.yaml.j2` (JobSet resource) and `workflow.yaml.j2` (Argo Workflow). Opening either file shows the exact YAML that gets submitted to Kubernetes, with Jinja2 expressions for dynamic values.

- **Computation stays in Python**: All helper functions (`_get_env`, `_build_affinity`, `_build_success_policy`, etc.) remain in `jobset.py`. The new `_build_role_context()` and `build_jobset_context()` functions assemble their results into a plain dict for template consumption.

- **`_normalize_env()` helper**: Converts the mixed `{name, value}` / `{name, valueFrom}` env var list to a discriminated form (`kind: value|fieldRef|secretKeyRef`) for clean template branching.

- **Two-pass rendering**: JobSet is rendered first (YAML string), then embedded into the Workflow template via `{{ step.jobset_yaml | indent(8) }}`. This matches the existing `yaml.safe_dump(js_manifest)` pattern.

- **`to_yaml_bool` filter**: Kubernetes API rejects Python `True`/`False`; the filter emits lowercase `true`/`false`.

- **`yaml.safe_load()` at submission**: The Kubernetes Python client requires a dict; rendered YAML is parsed back before the API call.

- **`create_jobset_manifest()` return type** changed from `tuple[str, dict]` to `tuple[str, str]` (js_name, rendered_yaml_string). Only caller was `_create_step_manifest()` in `launch_argo_workflow.py`.

**Removed dead code**: `_get_init_containers`, `_get_log_sidecar`, `_build_replicated_job` — their structure now lives in `jobset.yaml.j2`, their data computation in `_build_role_context()`.

**Files changed:**
- `pyproject.toml` — added `jinja2>=3.1.0`
- `src/seekr_chain/backends/argo/render.py` — new Jinja2 environment
- `src/seekr_chain/backends/argo/templates/jobset.yaml.j2` — JobSet template

**Gotcha — download-assets env vars**: `NODE_RANK` and `RESTART_ATTEMPT` must both be present in the `download-assets` init container. They are shell-expanded in `remote_md_path` (`job_index=${NODE_RANK}/attempt=${RESTART_ATTEMPT}/md.json`). Removing them causes `FileNotFoundError` at runtime when `parse_logs.py` reads `md.json` from S3.

**Gotcha — FileSystemLoader vs PackageLoader**: `jinja2.PackageLoader` fails with editable installs (`pip install -e .`). `FileSystemLoader(Path(__file__).parent / "templates")` works unconditionally and is the correct choice.

**Gotcha — unit test collection**: The hermetic k3d cluster fixture is session-scoped and autouse-adjacent — running `tests/unit/` requires passing `--real-cluster` to skip cluster setup, or pytest will attempt (and fail) to create a k3d cluster.

**Gotcha — Jinja2 dict attribute shadowing**: `expr.values` in a Jinja2 template resolves to `dict.values()` (the built-in method) because Jinja2 tries `getattr` before item access. Use `expr['values']` to force item lookup when a dict key shadows a built-in attribute name.

**Gotcha — data schema version vs package version**: The version file written to S3 was `__version__` (derived from git tags). On a fresh clone with no tags this is `0.0.0.xxx`, which broke parser routing. Replaced with `DATA_SCHEMA_VERSION = "1"` in `parse_logs.py` — a stable constant that only changes when the S3 directory layout changes.

**Known pre-existing bug**: `_build_failure_policy()` builds a `rules` list but never adds it to the returned policy dict. The template preserves this behavior (only `maxRestarts` is rendered).
